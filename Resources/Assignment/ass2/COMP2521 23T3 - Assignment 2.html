Task 1: Road Map ADT

Before doing any navigation, you will first need to collect information about nodes (intersections) and roads. Therefore, your first task is to implement a Road Map ADT which stores information about nodes and roads.

Like on Earth, roads on Zephyrus can be one-way or two-way, and only a small number of roads can be connected to each node (otherwise it would be impossible to control the traffic).

The Road Map ADT contains a large number of operations, but only some of them need to be implemented in Task 1. Here are the operations that you need to implement in Task 1:

Operation Signature/Description
RoadMapNew 
RoadMap RoadMapNew(int numNodes);
Creates a new road map with the given number of nodes (intersections) and no roads
RoadMapFree 
void RoadMapFree(RoadMap map);
Frees all memory allocated to the given map
RoadMapNumNodes 
int RoadMapNumNodes(RoadMap map);
Returns the number of nodes in the map
RoadMapNumRoads 
int RoadMapNumRoads(RoadMap map);
Returns the number of roads in the map
Note: Both one-way and two-way roads count as a single road
RoadMapAddRoad 
bool RoadMapAddRoad(RoadMap map, int node1, int node2,
                    bool isTwoWay, int travelMinutes);
Adds a road between nodes node1 and node2 that takes travelMinutes to drive down, if:
  • There is not already MAX_ROADS_PER_NODE roads connected to node1 or node2, and
  • There is not already a road (one-way or two-way) between node1 and node2
If isTwoWay is true, the new road should be a two-way road, otherwise it should be a one-way road from node1 to node2.
Returns true if the road was added, and false otherwise.
Assumes that node1 and node2 are valid and are not the same, and that travelMinutes is positive and less than MINUTES_PER_DAY (see time.h).
RoadMapGetRoadsFrom 
int RoadMapGetRoadsFrom(RoadMap map, int node, struct road roads[]);
Stores roads that can be used to drive away from node in the given roads array (in any order), and then returns the number of roads stored. In each of the roads stored, the fromNode field should be equal to node.

Example:
Suppose the road map contains the following roads:
  • Two-way road between 0 and 1
  • One-way road from 1 to 2
  • One-way road from 3 to 1
Then if RoadMapGetRoadsFrom is called with node 1, then two roads should be stored (in any order): a road from 1 to 0, and a road from 1 to 2.
RoadMapGetRoadsTo 
int RoadMapGetRoadsTo(RoadMap map, int node, struct road roads[]);
Stores roads that can be used to drive towards node in the given roads array (in any order), and then returns the number of roads stored. In each of the roads stored, the toNode field should be equal to node.

Task 2: Identifying Islands

Now that you've collected all the information you need, you're about to start working on the navigation feature of your app... when you make an important realisation! The population on Zephyrus is spread across contains several islands, some of which are fairly large. There is no way to drive between two islands, so if a user tries to find a route between two islands, you would like the app to respond quickly and tell the user that there is no route, instead of wasting time trying to find a route.

Therefore, you would like your app to pre-process the nodes and roads to identify islands. When a user wants to find a route between two nodes, the app will be able to check whether the two nodes are on the same island before calculating any routes.

Definition

Two nodes AA and BB are on the same island if and only if:

  • There is a road (either one-way or two-way) between AA and BB, or
  • There is a road (either one-way or two-way) between AA and a node which is on the same island as BB

In order to help achieve this, your task is to implement the following operations in the Road Map ADT. Note that you will need to modify some of your code from Task 1 to store information about islands (namely, your roadMap struct definition, RoadMapNew and RoadMapFree).

Operation Signature/Description
RoadMapProcessIslands 
void RoadMapProcessIslands(RoadMap map);
Internally processes and saves information about islands based on the current roads in the map.
Assumes that no roads will be added to the map after this function is called.

Note: This function returns no value, so calling this function alone will have no observable effect from a user's perspective. However, the purpose of this operation is to compute and store information in the map to enable the following operations to be efficient. All operations below assume that this function has been called first.
RoadMapNumIslands 
int RoadMapNumIslands(RoadMap map);
Returns the number of islands.
Note: This function must have a worst-case time complexity of O(1)O(1) with respect to the number of nodes.
RoadMapOnSameIsland 
bool RoadMapOnSameIsland(RoadMap map, int node1, int node2);
Returns true if node1 and node2 are on the same island, and false otherwise.
Note: This function must have a worst-case time complexity of O(1)O(1) with respect to the number of nodes.
RoadMapSetIslandName 
void RoadMapSetIslandName(RoadMap map, int node, char *name);
Given a node, sets the name of the island that the node is on to the given name. If the island was already given a name, then it should be replaced with the given name.
Note: This function must have a worst-case time complexity of O(1)O(1) with respect to the number of nodes.
RoadMapGetIslandName 
char *RoadMapGetIslandName(RoadMap map, int node);
Given a node, returns the name of the island that the node is on, or "(unnamed)" if the island has not been named via RoadMapSetIslandName.
Note: This function must have a worst-case time complexity of O(1)O(1) with respect to the number of nodes.

Note: This function should simply return the island name stored in the map instead of a copy of the name. The user must not modify or free the returned string.

Task 3: Navigation

You're finally ready to work on navigation!

Navigation should work in the same way as any other popular navigation app back on Earth (such as Google Maps) - users input a starting location, a destination, and a departure time, and the app finds the fastest route.

For more accuracy, you would also like your app to take traffic lights into account, as some nodes contain traffic lights. You know that the average amount of time spent waiting at a traffic light is TRAFFIC_LIGHT_WAIT minutes, so each traffic light in the middle of a route (not at the start or end of the route) will add this many minutes to the total travel time. If a node does not contain a traffic light, then there is no waiting time.

Your task consists of two parts. The first is to implement these simple operations in RoadMap.c. These may require you to modify some your existing code in RoadMap.c.

Operation Signature/Description
RoadMapAddTrafficLights 
void RoadMapAddTrafficLights(RoadMap map, int node);
Adds traffic lights to the given node. Does nothing if the node already has traffic lights.
RoadMapHasTrafficLights 
bool RoadMapHasTrafficLights(RoadMap map, int node);
Returns true if the given node has traffic lights, and false otherwise.

Your main task is to implement the following function in navigator.c:

struct route navigate(RoadMap map, int fromNode, int toNode,
                      struct time time);

This function should find and return the fastest route from fromNode to toNode.

time is the departure time, which consists of an hour and minute. The time struct is defined in time.h as follows:

struct time {
	int hour;   // between 0 and HOURS_PER_DAY - 1 (inclusive)
	int minute; // between 0 and MINUTES_PER_HOUR - 1 (inclusive)
};

The route struct is defined in navigator.h:

struct route {
	int fromNode;
	int toNode;
	struct time start;
	struct time end;
	int numLegs;
	struct leg *legs;
	int totalDuration; // in minutes
};

struct leg {
	int fromNode;
	int toNode;
	struct time start;
	struct time end;
	int duration; // in minutes
};

Each route struct contains:

Field(s) Description
fromNode and toNode These are the starting and finishing nodes of the route respectively, and should be the same as the fromNode and toNode arguments to the navigate function.
start and end These are the times at the start and end of the route respectively. In this task, start should be the same as the given departure time.
numLegs and legs Every route consists of one or more legs, where a leg involves driving down one road. If a route uses 3 roads, for example, then the number of legs is 3.
legs should point to a dynamically allocated array of leg structs, which contains the details of each leg of the route in order from start to finish, including:
  • The start and finishing nodes of the road
  • The time at the start of driving down the road, and the time at the end
  • The time spent on this leg. Note that this is the time spent driving down the road, and does not include the time spent waiting at a traffic light before driving on the road.
totalDuration This is the total amount of time (in minutes) taken by this route, including any waiting at nodes.

If there are multiple fastest routes from fromNode to toNode, then you may return any of them. If there is no route from fromNode to toNode, then you should return a route struct with numLegs set to 0 and legs set to NULL.

Task 4: Advanced Navigation

Dust storms occur all the time on Zephyrus! To help keep people safe, some roads are closed during certain periods of the day. Fortunately, you have learned that if a road is affected by dust storms, the dust storms will occur at the same time each day, so it is still possible to plan routes ahead of time.

If a road is affected by dust storms, people are only allowed to drive down the road if the road will be open the whole time they are on it. If this is not the case, they will have to wait at the start of the road until they are able to drive down the entire road, or use a different route.

For example, suppose a road takes 10 minutes to drive down, and is closed from 13:45 until 14:00. If it is 13:35, people are allowed to start driving down the road. But if it is 13:36, people are not allowed to start driving down the road, and they will either have to wait until 14:00, or take a different route. Once it is 14:00, people will be allowed to drive down the road again.

Due to the need to wait for dust storms to subside, it will sometimes save a driver more time if they start on their route later than they originally intended. For example, consider the following simple road map:

Suppose that a driver would like to go from 0 to 2, and would like to depart at 15:00. If they do depart at 15:00, they will reach node 1 at 15:10, but they would then have to wait until 15:40 before driving from node 1 to 2. They would therefore arrive at their destination at 15:55, and the duration of the route would be 55 minutes. Meanwhile, if the driver departed at 15:30, they would arrive at the exact same time, but would have spent much less time on the road (25 minutes).

Therefore, you would like your app to provide the route that not only arrives as early as possible, but that has the shortest duration. This means the starting time of the route may be later than the inputted departure time.

Your task consists of two parts. The first is to implement the following function in RoadMap.c: 

void RoadMapSetClosedTimes(RoadMap map, int node1, int node2,
                           struct time closedFrom,
                           struct time closedUntil);

This function specifies that the road between node1 and node2 should be closed from closedFrom until closedUntil each day.

You will also need to make sure that the road structs stored by RoadMapGetRoadsFrom and RoadMapGetRoadsTo contain these times, if the road is affected by dust storms. This may require you to modify your RoadMapGetRoadsFrom and RoadMapGetRoadsTo functions to store these times. If a road is not affected by dust storms, you should ensure that closedFrom and closedUntil are equal.

Your main task is to implement the following function in navigator.c:

struct route advancedNavigate(RoadMap map, int fromNode, int toNode,
                              struct time time);

This function should find and return the route from fromNode to toNode that arrives at toNode as soon as possible after time, and that departs as late as possible (i.e., has the shortest duration).

If there are multiple such routes, then you may return any of them. If there is no route from fromNode to toNode, then you should return a route struct with numLegs set to 0 and legs set to NULL.

To earn full marks for this task, your solution must not use the brute-force method of trying multiple departure times until you find the departure time that results in the shortest-duration route. Solutions that do this will earn at most 10 out of 15 marks.

  • Suppose the driver departs at 11:45.

  • The soonest they will be able to reach node 5 is 11:55 (0 - 1 - 5), and it would be 11:56 after waiting for the light. Since there is 9 minutes before the road from 5 to 6 closes and the road takes 10 minutes to traverse, they would need to wait until 12:15 before driving.

Frequently Asked Questions

  • Are we allowed to create our own functions? You are always allowed to create your own functions. All additional functions you create should be made static.
  • Are we allowed to create our own #defines and structs? Yes.
  • Are we allowed to #include any other libraries? No. All the libraries required for this assignment are provided already.
  • Are we allowed to change the signatures of the given functions? No. If you change these, your code won't compile and we won't be able to test it.
  • What errors do we need to handle? You should handle common errors such as NULL returned from malloc by printing an error message to stderr and terminating the program. You are not required to handle other errors.
  • What invalid inputs do we need to handle? You are not required to handle invalid inputs, such as NULL being passed in as a road map, an integer outside the range [0,N−1][0,N−1] being passed in as a node (where NN is the number of nodes). It is the user's responsibility to provide valid inputs.
  • Will we be assessed on our tests? No. You will not be submitting any test files, and therefore you will not be assessed on your tests.
  • Are we allowed to share tests? No. Testing is an important part of software development. Students that test their code more will likely have more correct code, so to ensure fairness, each student should independently develop their own tests.
